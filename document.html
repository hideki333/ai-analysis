新規 app/Services/AiAnalysisService.php ▼

役割

このシステムのコアロジック。APIへのリクエスト送信からレスポンス検証、DBへの保存まで一貫して担当します。

主なメソッド

  • analyze(string $imagePath, string $mockMode) ― 外部またはモックAPIを呼び出し、検証後にDBへ保存
  • validateResponse(?array $body) ― API仕様に沿ったレスポンス構造を検証。異常があればエラー理由を返す
  • callApi(string $imagePath, string $mockMode) ― HTTPリクエストを送信（Docker環境など、マルチプロセス環境で使用）。HTTPエラー時は例外をスロー
  • callMockDirectly(string $mode) ― モックレスポンスを直接生成（HTTP通信なし）

ポイント

  • php artisan serve はシングルスレッドのため、自分自身へHTTPリクエストを送るとデッドロックになる
  • AI_USE_MOCK=true（ローカル開発）：HTTP通信せず callMockDirectly() を直接呼び出してデッドロックを回避
  • AI_USE_MOCK=false（Docker環境）：callApi() でHTTPリクエストを送信。Nginx経由でモックAPIにアクセス
  • 本番API切り替えは .env の AI_API_URL を変更し、AI_USE_MOCK=false にするだけ

役割

UI画面に関わるリクエストを受け付けるコントローラ。ロジックは持たず AiAnalysisService に委譲します。

メソッド

新規 app/Http/Controllers/MockApiController.php ▼

役割

存在しない外部AI分析APIを模倣する仮想エンドポイントです。POST /api/mock/analyze でHTTPからも呼び出せます。

レスポンス仕様

  • mock_result=success：必ず成功レスポンスを返す。class は 1〜5 のランダム整数、confidence は 0.5000〜0.9999 のランダム小数
  • mock_result=failure：必ず失敗レスポンスを返す
  • mock_result=invalid：仕様外レスポンス（estimated_data 欠落）を返す。バリデーション検証用

注意

  • Docker環境では http://nginx/api/mock/analyze でHTTP経由アクセス。ローカルでは AiAnalysisService が直接メソッドを呼び出す（デッドロック回避）
新規 app/Models/AiAnalysisLog.php ▼

役割

ai_analysis_log テーブルに対応するEloquentモデル。

設定

  • $timestamps = false ― Laravelデフォルトの created_at/updated_at は使わず、仕様のカラム（request_timestamp / response_timestamp）を使用
  • $fillable ― 一括代入を許可するカラムを明示的に定義（マスアサインメント保護）
  • $casts ― DBの値を適切なPHP型に自動変換（boolean, integer, decimal, datetime）
Migration database/migrations/****_create_ai_analysis_log_table.php ▼

役割

課題で指定されたMySQLのCREATE文をLaravelマイグレーション形式に変換したものです。SQLite・MySQL双方で動作します。

MySQLとの主な違い

  • tinyint(1) → $table->boolean()
  • decimal(5,4) → $table->decimal('confidence', 5, 4)
  • datetime(6) → $table->dateTime('...', 6)（マイクロ秒対応）
  • AUTO_INCREMENT の id → $table->id()

新規 docker-compose.yml / docker/php/Dockerfile / docker/nginx/default.conf ▼

役割

Docker環境でシステムを起動するための設定ファイル群。Nginx（リバースプロキシ）・PHP-FPM（アプリ実行）・MySQL（DB）の3コンテナ構成です。

ポイント

  • phpコンテナの環境変数 AI_API_URL=http://nginx/api/mock/analyze でコンテナ間通信を実現
  • phpコンテナの環境変数 AI_USE_MOCK=false でHTTP通信モードに切り替え
  • http://localhost:8000 でブラウザからアクセス可能（ポート8000:80でNginxに転送）
  • MySQLデータは mysql_data Volumeで永続化
変更 .env / config/services.php ▼

追加した設定

  • AI_API_URL ― APIのエンドポイントURL。本番APIへの切り替え時に変更する（ローカルは http://localhost:8000/api/mock/analyze、Docker内部は http://nginx/api/mock/analyze）
  • AI_USE_MOCK ― true のとき HTTP通信せず直接モックメソッドを呼び出す（ローカル用）。false のとき AI_API_URL にHTTPリクエストを送信（Docker用）
  • config('services.ai_analysis.url') / config('services.ai_analysis.use_mock') ― .env の値をServiceから参照するためのブリッジ
DBスキーマ（ai_analysis_log）
カラム名 型 NULL 説明
id PK int AUTO_INCREMENT NO レコードの一意識別子
image_path varchar(255) YES リクエストに使用した画像ファイルパス
success tinyint(1) NO APIが成功を返したか（1=成功 / 0=失敗）
message varchar(255) YES APIのメッセージ（成功時は"success"、失敗・異常時はエラー内容）
class int YES AIが判定した分類クラス番号（success=true のときのみ格納）
confidence decimal(5,4) YES 判定の信頼度スコア 0〜1（success=true のときのみ格納）
request_timestamp datetime(6) YES APIリクエストを送信した日時（マイクロ秒精度）
response_timestamp datetime(6) YES APIレスポンスを受信した日時（マイクロ秒精度）
APIレスポンスパターンと保存処理
Success（正常成功）
{
  "success": true,
  "message": "success",
  "estimated_data": {
    "class": 3,
    "confidence": 0.8683
  }
}
→ class・confidence をDBに保存
Failure（API側の失敗）
{
  "success": false,
  "message": "Analysis failed due to
    an internal error."
}
→ success=false、messageをDBに保存
異常レスポンス（仕様外）
{
  "success": true,
  "message": "success"
  // estimated_data が欠落
}
→ validateResponse() が検出し
success=false・「不正なレスポンス:〜」で保存
接続エラー / 非JSONレスポンス
// タイムアウト・HTTP 5xx・
// HTMLエラーページなど
→ catch ブロックで捕捉し
success=false・「API接続エラー:〜」で保存
レスポンス検証ルール（validateResponse）
チェック対象 検証内容 異常時のmessage
レスポンスボディ全体 null でないこと（有効なJSON） レスポンスがJSONではありません
success フィールドが存在し、bool型であること 'success' フィールドがbool型で存在しません
message フィールドが存在し、文字列であること 'message' フィールドが文字列で存在しません
estimated_data success=true のとき、配列で存在すること success=true なのに 'estimated_data' が存在しません
estimated_data.class success=true のとき、整数で存在すること 'estimated_data.class' が整数で存在しません
estimated_data.confidence success=true のとき、0〜1の数値で存在すること 'estimated_data.confidence' が0〜1の数値で存在しません
起動・確認方法

【A】ローカル開発（php artisan serve）

1
依存関係インストール＆セットアップ
cd {プロジェクトパス}/ai-analysis
composer install
php artisan key:generate
php artisan migrate
2
開発サーバーを起動
php artisan serve

※ .env の AI_USE_MOCK=true でモックを直接呼び出し（デッドロック回避）

3
ブラウザでアクセス
http://localhost:8000

【B】Docker環境

1
コンテナをビルド＆起動
cd {プロジェクトパス}/ai-analysis
docker compose up -d --build
2
Composerインストール＆マイグレーション
docker compose exec php composer install
docker compose exec php php artisan key:generate
docker compose exec php php artisan migrate
3
ブラウザでアクセス
http://localhost:8000

※ docker-compose.yml で AI_USE_MOCK=false に設定済み。Nginx経由でモックAPIへHTTP通信

4
コンテナの停止
docker compose down

【共通】本番API → モックAPIに切り替える場合

1
ローカル（php artisan serve）：.env を変更
AI_USE_MOCK=true
AI_API_URL=http://localhost:8000/api/mock/analyze
2
Docker環境：docker-compose.yml の environment を変更し再起動
php:
  environment:
    AI_USE_MOCK: "true"
    AI_API_URL: http://nginx/api/mock/analyze

【共通】モックAPI → 本番APIに切り替える場合

1
ローカル（php artisan serve）：.env を変更
AI_USE_MOCK=false
AI_API_URL=https://example.com/
2
Docker環境：docker-compose.yml の environment を変更し再起動
php:
  environment:
    AI_API_URL: https://example.com/
    AI_USE_MOCK: "false"
3
モックAPIを直接テストする場合（curl）
curl -X POST http://localhost:8000/api/mock/analyze \
  -H "Content-Type: application/json" \
  -d '{"image_path":"/image/test.jpg","mock_result":"success"}'
